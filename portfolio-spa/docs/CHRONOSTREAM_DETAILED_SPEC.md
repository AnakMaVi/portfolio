# ChronoStream Detailed Technical Specification

## 1. Project overview
ChronoStream is a high-performance in-browser cryptographic engine built with Rust and compiled to native WebAssembly (WASM). The objective is to demonstrate deterministic CPU execution in the recruiter browser with minimal runtime overhead and explicit performance telemetry.

This module is integrated into the React portfolio as a live demo and exposes native functions for encryption and SHA-256 hashing.

## 2. Goals
- Execute cryptographic primitives fully in browser CPU.
- Avoid backend dependency for demo execution.
- Offer measurable latency in microseconds for transparent benchmarking.
- Provide a clean integration boundary between React and WASM exports.

## 3. Scope
### In scope
- Rust crate compiled to WASM via wasm-pack.
- Exported functions through wasm-bindgen.
- React component with benchmark and visualization.
- Dockerized development build pipeline.

### Out of scope
- Enterprise-grade encryption suite (AES-GCM, key derivation, key vault).
- Server-side key management.
- Compliance certification (FIPS, SOC2, ISO 27001).

## 4. Technology stack
- Rust 1.96+ (toolchain)
- wasm-bindgen
- wasm-pack
- sha2 crate for SHA-256 digest
- base64 crate for transport-safe encrypted payload output
- React + Vite for UI integration
- Docker for reproducible development build

## 5. Architecture
### 5.1 High-level flow
1. React loads the generated wasm JS loader asynchronously.
2. Loader initializes the `.wasm` binary in browser runtime.
3. UI calls exported Rust functions (`encrypt_data`, `generate_hash`).
4. Results return to JS as plain strings.
5. Benchmark layer measures latency and renders p50/p95 chart.

### 5.2 Runtime boundaries
- Native compute: Rust/WASM module.
- Orchestration and rendering: React/JS.
- Data never leaves browser unless user explicitly copies/exports.

## 6. Rust module contract
### 6.1 Crate location
- `chronostream-wasm/src/lib.rs`
- `chronostream-wasm/Cargo.toml`

### 6.2 Exported APIs
#### `encrypt_data(input: &str, key: &str) -> Result<String, JsValue>`
- Performs symmetric XOR encryption over UTF-8 bytes.
- Validates non-empty key.
- Encodes encrypted bytes using Base64 (no padding).
- Returns a JS-friendly string or explicit error.

#### `generate_hash(input: &str) -> String`
- Computes SHA-256 digest for the full payload.
- Processes large input in chunks to keep memory stable.
- Returns lowercase hex digest.

### 6.3 Error model
- Invalid key produces an explicit JS exception message.
- Hash function is non-fallible for standard string input.

## 7. WASM build artifacts
Generated by wasm-pack in `chronostream-wasm/pkg`:
- `chronostream_wasm.js` (loader + bindings)
- `chronostream_wasm_bg.wasm` (compiled binary)
- Type declaration files

These artifacts can be consumed directly by Vite and bundled in frontend output.

## 8. React integration details
### 8.1 Component
- `src/components/WasmCryptor.jsx`

### 8.2 Functional features
- Async module initialization with user feedback.
- Encryption panel with input + secret key.
- SHA-256 panel for massive text hashing.
- Single-run latency metrics (microseconds):
  - WASM encrypt latency
  - JS baseline encrypt latency
  - Ratio JS/WASM
- Benchmark mode (100 iterations):
  - p50 latency (WASM + JS)
  - p95 latency (WASM + JS)
  - speedup ratios for p50/p95
- Mini chart visualization for benchmark output.
- Copy actions:
  - Copy encrypted payload
  - Copy hash output

### 8.3 Benchmark methodology
- Uses `performance.now()` timing in browser.
- Converts milliseconds to microseconds by multiplying by 1000.
- Stores per-iteration arrays.
- Computes percentile via linear interpolation over sorted latency values.

## 9. Dockerized development build
### 9.1 Dockerfile location
- `chronostream-wasm/Dockerfile`

### 9.2 Container responsibilities
- Install Rust target `wasm32-unknown-unknown`.
- Install `wasm-pack` automatically.
- Install `binaryen` and run `wasm-opt -Oz` for size optimization.
- Compile source mounted by volume and output `pkg/`.

### 9.3 Recommended usage
Build image:
```powershell
docker build -t chronostream-wasm-dev c:\htdocs\portfolio\portfolio-spa\chronostream-wasm
```

Generate pkg in module folder:
```powershell
docker run --rm -v c:\htdocs\portfolio\portfolio-spa\chronostream-wasm:/work chronostream-wasm-dev
```

Generate directly in React public assets:
```powershell
docker run --rm -v c:\htdocs\portfolio\portfolio-spa\chronostream-wasm:/work -v c:\htdocs\portfolio\portfolio-spa\public\chronostream:/out -e WASM_OUT_DIR=/out chronostream-wasm-dev
```

## 10. Performance notes
- WASM startup has one-time initialization cost.
- For tiny payloads, JS and WASM can be close due to call overhead.
- For repeated workloads and larger payloads, WASM generally benefits from native compiled execution patterns.
- p50/p95 is preferred over a single sample to avoid noisy claims.

## 11. Security considerations
### 11.1 Current demo state
- XOR encryption is intentionally basic and educational.
- Not suitable for production-grade confidentiality requirements.

### 11.2 Production upgrade path
- Replace XOR with audited symmetric cipher (AES-GCM or ChaCha20-Poly1305).
- Add nonce management strategy.
- Add key derivation (Argon2id / HKDF depending on context).
- Add authenticated encryption and integrity verification.

### 11.3 Browser constraints
- Secret key lives in client memory while processing.
- Avoid logging secrets in console or telemetry.
- Use strict CSP and SRI in production hosting.

## 12. Testing strategy
### 12.1 Unit tests (Rust)
Recommended tests:
- Empty key rejects encryption.
- Same input/key produces deterministic encrypted output.
- Hash output length is exactly 64 hex chars.
- Hash known vector matches expected digest.

### 12.2 Integration tests (Frontend)
Recommended tests:
- WASM module load success/failure states.
- Encryption action updates output and latency cards.
- Benchmark computes valid p50 <= p95.
- Copy buttons write expected text.

## 14. Troubleshooting
### Symptom: `cargo` not recognized
- Add `C:\Users\<user>\.cargo\bin` to PATH.
- Restart terminal.

### Symptom: wasm-pack install blocked by policy
- Use precompiled wasm-pack binary install.
- Request allowlist for `%USERPROFILE%\.cargo\bin` and temp cargo build path.

### Symptom: wasm import fails in React
- Ensure `pkg/` exists and contains `.js` + `.wasm`.
- Ensure import path points to generated package.
